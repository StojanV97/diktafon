# Diktafon — Project Reference

Audio recording + transcription journal app with Serbian UI.
**Stack**: Expo/React Native mobile (JS) — fully standalone, no backend required.
**Purpose**: Record audio → transcribe (on-device whisper.rn or direct AssemblyAI cloud) → browse & share transcripts.

---

## Quick Commands
```
cd mobile && npx expo run:ios           # Dev build on iOS simulator
cd mobile && npx expo run:android       # Dev build on Android
cd mobile && npx expo prebuild --clean  # Clean native rebuild (after plugin/config changes)
cd mobile && npx expo start             # Start Metro bundler (dev client must already be installed)
```

## Gotchas
- Requires Expo dev builds, NOT Expo Go. Never suggest `npx expo start --go`.
- whisper.rn has native code: any change to app.json plugins or expo-build-properties needs `npx expo prebuild --clean` before `run:ios`.
- `ios/` and `android/` are generated by prebuild and gitignored. Never commit them.
- All user-facing strings are in Serbian. Do not translate to English. Use Serbian for any new UI text.
- journalStorage.js uses atomic writes (.tmp → .bak → move). Maintain this pattern for new file-writing code.
- Audio files are .m4a (recorded via expo-audio). whisper.rn accepts this format directly.
- No test suite exists. Do not suggest running tests or create test files unless explicitly asked.
- No ESLint/Prettier configured. Follow existing code style (no semicolons in some files, 2-space indent).

## Coding Rules
- Use theme.js constants for all colors, spacing, radii, typography. Never hardcode style values.
- New screens must be added to the Stack navigator in App.js.
- New services should follow the facade pattern: implementation in a dedicated service file, thin wrapper in journalApi.js.
- State management is local (useState/useEffect + AsyncStorage). No Redux, no Context providers.
- File storage ops must go through journalStorage.js. Never write directly to DocumentsDir/journal/.

### Services

**whisperService.js** — On-device transcription using whisper.rn (whisper.cpp bindings):
- Model: `ggml-base.bin` (~141MB), downloaded from HuggingFace on first use
- Cached at `DocumentsDir/whisper-models/ggml-base.bin`
- Serbian diacritics prompt hardcoded (č, ć, š, ž, đ)
- Exports: `getModelStatus()`, `downloadModel(onProgress)`, `deleteModel()`, `transcribe(audioFileUri, onProgress)`, `releaseContext()`
- Singleton whisperContext — initialized once, released on app background

**assemblyAIService.js** — Direct AssemblyAI REST API calls:
- API key stored in expo-secure-store
- Upload via `expo-file-system` uploadAsync → create transcript job → poll status
- Speaker diarization with `[Govornik A – 0:00]` formatting
- Exports: `getApiKey()`, `setApiKey()`, `removeApiKey()`, `hasApiKey()`, `submitAndGetId(fileUri)`, `checkTranscript(transcriptId)`

**journalApi.js** — Thin facade maintaining same API shape:
- `transcribeLocal(fileUri, filename, onProgress)` → whisperService.transcribe
- `submitAssemblyAI(fileUri, filename)` → assemblyAIService.submitAndGetId
- `checkAssemblyAI(assemblyaiId)` → assemblyAIService.checkTranscript

**journalStorage.js** — local file-based storage using expo-file-system:
```
DocumentsDir/journal/
├── folders.json      # Folder metadata array
├── entries.json      # Entry metadata array
├── audio/            # {entryId}.m4a files
└── texts/            # journal_{entryId}.txt files
```
Key ops: `createFolder`, `fetchFolders`, `updateFolder`, `deleteFolder` (cascades),
`createEntry`, `fetchEntries`, `fetchEntry`, `deleteEntry`,
`updateEntryToProcessing`, `completeEntry`, `failEntry`, `entryAudioUri`, `getAllTags`,
`getOrCreateDailyLogFolder`, `createDailyLogEntry`, `fetchDailyLogEntries`,
`getDailyCombinedTranscript`, `moveEntryToFolder`
Migration: resets stuck "processing" entries (>24h) back to "recorded"

### Screens

**DirectoryHomeScreen**: Folder list with FAB to create. Settings gear icon in header. "Danas" card for daily log. Each folder card shows color accent, name, date, tags.

**DirectoryScreen**: Entry list + recording interface. FAB.Group with mic (record) and file upload. Engine choice dialog with model download/API key pre-flight checks. Polls processing entries every 5s.

**DailyLogScreen**: Daily log with section-by-section view grouped by date. Batch transcription, combined transcript view with copy/share. Same engine dialog with pre-flight checks.

**EntryScreen**: Full transcript viewer + audio player. Seekable progress bar, rewind/ff 10s, play/pause. Bottom bar: copy text, share text, save audio/transcript to Files.

**SettingsScreen**: Three sections:
1. Whisper Model — download/delete status, progress bar
2. AssemblyAI — API key input (secure store), save/delete
3. Default Engine — radio: "Na uredjaju" vs "AssemblyAI"

### Transcription Flow
1. Tap mic FAB → request permission → record → waveform overlay
2. Stop → copy audio to `journal/audio/{id}.m4a` → `createEntry()` with status `recorded`
3. Tap transcribe → engine choice dialog (default from Settings)
4. **On-device**: Check model downloaded (auto-download if not) → `whisperService.transcribe()` → `completeEntry()` → done
5. **AssemblyAI**: Check API key → `assemblyAIService.submitAndGetId()` → `updateEntryToProcessing()` → poll every 5s → `completeEntry()`/`failEntry()`

## Data Flow Patterns

1. **Record**: expo-audio → stop → copy to `journal/audio/` → create entry (status: `recorded`)
2. **Transcribe on-device**: whisper.rn with ggml-base model → text with Serbian diacritics → `completeEntry()` → write `.txt` + update JSON
3. **Transcribe AssemblyAI**: Upload to api.assemblyai.com → create transcript → poll status → speaker-formatted text → complete/fail
4. **Local storage**: JSON metadata (truncated 200-char text) + separate `.txt` for full text (keeps JSON small)
5. **Cascading deletes**: folder → all entries → audio files + text files
